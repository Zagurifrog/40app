🎂 Birthday project retrospective

How We Built
Tal's 40th Birthday
AI Photo App

A single-file web app that lets party guests generate an AI photo of themselves with Tal across any decade — from the 1920s to 2050.

See the live app →
🎨
Step 1 — Design
UI designed in Google Stitch

The entire visual design — dark cyberpunk aesthetic, neon accents, Hebrew RTL layout, decade tiles, and the result screen — was iterated and finalized using Google Stitch before writing a single line of code.

Stitch let us rapidly try different color schemes, card layouts, and typography until the design felt right: dark background, neon glow effects, 9:16 Instagram Story output format.

Starting in a design tool means the HTML build is just implementation — no layout guessing mid-code.
Dark cyberpunk theme Hebrew RTL Rubik font 9:16 Story format Neon glow system
🏗️
Step 2 — Build
Single-file HTML app

The full app lives in one index.html — no build tools, no npm, no server. Tailwind CSS via CDN, inline JS, and Google Fonts. Guests can use it directly from GitHub Pages.

Tal's portrait was encoded as Base64 and embedded directly in the file, so every generated image always includes her — no external assets to load or lose.

Zero infrastructure = zero failure points at a party. One URL, works offline after first load.
Tailwind CDN FileReader API Base64 photo embed GitHub Pages
🔑
Step 3 — API
Connecting Gemini Image Generation

Used Gemini 2.5 Flash Image via Google AI Studio. The app sends two images (Tal's photo + guest's photo) plus a decade-specific prompt, and receives a generated composite image back.

Model selection took a few attempts — gemini-2.0-flash-preview-image-generation was deprecated, then gemini-2.0-flash-exp-image-generation, finally landing on gemini-2.5-flash-image.

Gemini's image generation can take both photos as visual reference, making the output feel personal to each guest.
🐛
Step 4 — Debug marathon
Everything that broke (and why)
🔄 Placeholder replaced twice
The validation check if (key === "PASTE...") also contained the placeholder — both got replaced with the real value, so the check always fired.
🏷️ Wrong model name
Google renamed the model mid-session. Had to query the live API for the actual model list to find gemini-2.5-flash-image.
🔑 API key leaked
GitHub's secret scanner found the key in the public repo within minutes and invalidated it. Twice.
🐪 inlineData vs inline_data
The API response used camelCase inlineData but the parser looked for snake_case inline_data — silent miss, no image.
🚫 IMAGE_OTHER block
Gemini refused to generate photorealistic images of real identifiable people. Added a friendly error message for guests.
📦 Images too large
Tal's embedded photo was 440KB — too many tokens. Added client-side resize to 768px before sending.
🛡️
Step 5 — Security
Cloudflare Worker as API proxy

After the API key kept getting leaked from the public repo, we set up a Cloudflare Worker — a free serverless function that sits between the app and Gemini. The key lives as a secret environment variable in Cloudflare, never in the code.

The app now calls gemini-proxy.zaguri.workers.dev instead of Google directly. The worker adds the key and forwards the request.

Any key in a public GitHub repo gets found and invalidated by automated scanners within minutes. A proxy is the only real fix for a client-side app.
